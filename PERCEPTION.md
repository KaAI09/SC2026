# 차선 인지 (Perception)

확정된 단일 인지 파이프라인. 카메라 한 장을 **캘리브레이션된 metric BEV**로 펴서 차선을
검출·분류하고 자차 통로 중앙선을 추정해 `/lane/state`(center_error·heading·confidence·…)로
발행한다. 실험용 프리셋(구 G1~G6)·모드는 없다 — 파이프라인은 하나다.
`camera:=''`면 front-view 레거시 경로로 폴백한다(§6 한계 적용).

- 코어(순수, ROS 무관): [dracer_core/perception_core.py](D-Racer-Kit/src/dracer_core/dracer_core/perception_core.py)
- 온라인 노드: [perception/perception_node.py](D-Racer-Kit/src/perception/perception/perception_node.py) — `/camera/image/compressed` → `/lane/state` (+ `/lane/debug/compressed` 디버그 오버레이)
- 오프라인 시각화/실험: [offline/perception_probe.py](offline/perception_probe.py) (BEV 실험 포함)

## 1. 흐름

```
frame ─► ① 색 마스크 ─► ② BEV 워프 ─► ③ morphology·색 게이트 ─► ④ sliding-window 다차선
         (지면 밴드만)   (LUT = 캘리브된 크롭)                            │
        ⑧ state ◄─ ⑦ Tracker ◄─ ⑥ ego 중앙선 ◄─ ⑤ 7-label 분류
```

1. **색 마스크** — HSV 흰/노랑 임계. 원본 프레임에서, 단 **BEV가 실제로 읽는 행(`cam.src_rows`)만**. 그 위쪽 행은 어떤 BEV 셀에도 기여하지 않는다.
2. **BEV 워프** — 1채널 마스크 2장을 warp(3채널 프레임이 아니라). 색은 per-pixel 판정이라 리샘플 **전에** 임계하는 게 더 싸고 더 선명하다(보간된 중간색이 HSV 게이트를 속이지 않음).
3. **morphology·게이트** — **워프 이후**. 여기서는 1픽셀이 곧 1거리라 커널·면적 임계가 26cm에서도 78cm에서도 같은 뜻을 가진다. NEAREST 워프가 남긴 행 복제도 함께 메운다.
4. **sliding-window** — 하단 히스토그램 base → 곡률(방향 EMA) 따라 창을 위로 쌓아 차선 궤적 추출
5. **분류** — 색(흰/노랑) + 위치(base x) + 방향(heading·곡률)으로 7라벨: `W-L/R`, `YS/YL/YR-L/R`
6. **ego 중앙선** — 인접한 좌/우 경계 쌍의 중앙선; 쌍이 없으면 단일 차선에서 차선폭 절반만큼 coast
7. **Tracker** — ego 좌/우 계수 EMA + 차선폭 유지 + turn
8. **state** — center_error·ema·heading·confidence·curvature·state·used_fallback

`camera:=''`(캘리브 없음)이면 **front-view 레거시 경로**로 떨어진다 — 사다리꼴 ROI를 쓰고,
morphology가 워프 전에 돌며, 모든 임계가 화면 비율의 의미를 갖는다(§6 한계 그대로).

## 2. 기법

| 단계 | 기법 | 요점 |
|---|---|---|
| 색 마스크 | HSV 색 마스크 | 지면 밴드만 처리(cvtColor 픽셀 −38%) |
| BEV 워프 | 캘리브된 remap LUT | **LUT 자체가 정확한 사다리꼴 크롭이다.** 손으로 튜닝한 `roi_*` 사다리꼴은 BEV 모드에서 제거됨(§7.8) |
| morphology·게이트 | 세로 CLOSE + 색 지배 게이트 | 워프 후 → 커널이 **물리 길이**(`morph_cm`). 소수 색 비율 < `color_gate`면 그 색 통째 제거 |
| sliding-window | base 히스토그램 → 방향 EMA 곡선추종 | 점선도 창이 간격 건너뛰며 이어줌. 세로 span 미달(정지선 등)은 탈락 |
| 병합 | **폴리라인 근접(MAX \|Δx\|)** | 두 스택이 전 구간 일치(max<`merge_dx`)면 같은 차선. **한쪽 끝이라도 벌어지면 병합 거부** → Y분기·원근수렴쌍 보존 |
| 분류 | heading 우선 + 곡률 보조 | 노랑 곡률로 직진S/좌L/우R |
| 쌍매칭 | y겹침 + 최소간격 게이트 | 교차(간격 붕괴)·비평행 쌍 기각 |
| coast | 단일 차선 ± 차선폭/2 평행이동 | **관측 y-span으로 클램프**(외삽 발산 차단). 고정폭이라 정확도 한계 → BEV로 근본 개선 예정 |
| 안정화 | center_error EMA + 상태 명명 | OK/LOW_CONF/OUTLIER/HOLD/LOST |

## 3. 좌표계 — 캘리브레이션 BEV (기본), front-view (폴백)

`camera.yaml`이 주어지면 차선 검출은 **metric BEV** 위에서 돈다. 1픽셀 = 1/`px_per_cm` cm이므로
차선폭·쌍매칭·coast·heading이 전부 **실제 거리**가 된다. 그리기는 `bev_coeffs_to_image()`로
원본 카메라 뷰에 역워프해 사람이 읽을 수 있게 한다. `camera:=''`면 front-view 레거시 경로로
떨어지고 §6의 한계 (a)~(d)가 그대로 살아난다.

**런타임 해상도는 자유 변수다.** `camera.yaml`은 캘리브한 해상도 그대로 저장하고,
perception이 카메라가 **실제로 보내는** 크기에 맞춰 `CameraModel.match()`로 정확 rescale한다
(GStreamer 경로가 순수 videoscale이라 오차 0). 둘이 어긋나면 `to_bev()`가 **예외를 던진다** —
예전에는 조용히 검게 채웠고, 그게 §7.8의 함정이었다.

**차선폭 35cm의 근거**: 실측이 내부 32cm / 외부 38cm(테이프 폭 3cm)인데, 검출 다항식은
테이프의 **중심선**을 따라간다(`sliding_window_lanes`가 창 내 픽셀 x를 평균) → 코드가 쓰는
차선폭은 **중심선 간 35cm**다.

## 4. 파라미터 · 실시간 튜닝

`perception_core.Cfg`의 모든 필드가 perception_node의 **live ROS 파라미터**다.
profile `[perception]` 섹션이 seed하고, 실행 중 다음처럼 즉시 반영된다(파이프라인
재생성):

```bash
ros2 param set /perception_node lane_width_cm 35.0
ros2 param set /perception_node merge_dx_cm 5.0
ros2 param set /perception_node colors "['white']"
```

**BEV 모드에서는 cm 파라미터만 만진다.** `cfg_to_px()`가 이들을 BEV 픽셀로 한 번에 해석하고,
그 아래 모든 단계는 평범한 픽셀을 계속 받는다. px 필드(`sw_margin`, `sw_minpix`, …)를 직접
set 하면 다음 `cfg_to_px()` 호출에 덮어써진다.

| cm 파라미터 | 뜻 | 대응 px |
|---|---|---|
| `lane_width_cm` 35.0 | 트랙 차선폭(중심선 간) | `lane_width_default` |
| `sw_margin_cm` 5.0 | 슬라이딩 창 반폭 | `sw_margin` |
| `sw_peak_sep_cm` 15.0 | base peak 최소 분리 | `sw_peak_sep` |
| `merge_dx_cm` 5.0 | 같은 테이프로 병합할 근접도 | `merge_dx` |
| `pair_gap_min_cm` 8.0 | 쌍 붕괴(교차) 기각 | `pair_gap_min` |
| `jump_max_cm` 15.0 | 프레임 간 최대 차선 점프 | `jump_max` |
| `heading_cm` 5.0 | 회전 판정 실제 횡드리프트 | `heading_frac` |
| `morph_cm` 1.25 | 세로 CLOSE 커널 | `morph_v` |
| `sw_minpix_cm2` 1.125 | 창 내 최소 차선 **면적** | `sw_minpix` |
| `sw_peak_min_cm` 2.5 | 히스토그램 peak 최소 **길이** | `sw_peak_min` |
| `gate_min_cm2` 5.0 | 색 게이트 발동 최소 **면적** | `gate_min_px` |

마지막 넷이 마지막까지 남아 있던 **스케일 함정**이었다. 픽셀 *개수*는 면적이라 `px_per_cm`의
**제곱**으로 변한다 — BEV 스케일을 절반으로 낮추면 모든 카운트가 1/4이 되는데 임계값은 그대로라
검출이 로그 한 줄 없이 죽는다. 면적·길이로 표현하면 면역이다. 기본값은 `px_per_cm=4`에서
기존 px 값(5/18/10/80)과 **정확히 일치**하도록 잡아 기존 튜닝을 보존한다.

**프레임 수 기준 파라미터는 Hz에 묶여 있다.** `lost_reset`·`lost_stop_frames`·`outlier_relatch`·
`median_window`·`ema_alpha`는 전부 시간이 아니라 **프레임 수**다. 인지가 10.7 → 30Hz가 되면서
같은 값의 실제 지속시간이 **1/3로 줄었다.** Hz를 바꾸면 이들도 같이 봐야 한다.

| 안정화 파라미터 | 기본 | @30Hz | 뜻 |
|---|---|---|---|
| `outlier_jump` | 0.5 | — | 이만큼 튀면 일단 거부 |
| `outlier_relatch` | 6 | 0.2s | **연속 거부가 이만큼이면 사실로 받아들이고 재시드** (§7.10) |
| `lost_stop_frames` | 8 | 0.27s | 검출 없음이 이만큼 → `LOST` |
| `lost_reset` | 8 | 0.27s | 트래커가 좌/우를 버리는 시점 (coast 중엔 발동 안 함) |

디버그 4패널은 **누가 구독할 때만** 만든다(§7.8) — 튜닝 중에는 rqt를 붙이거나 recorder를
`image_topic:=/lane/debug/compressed`로 돌리면 자동으로 켜진다. 주행 raw로 패널을 되살리려면
[offline/panel_replay.py](offline/panel_replay.py).

## 5. state 계약 (LaneState)

| 필드 | 의미 | 누가 쓰나 |
|---|---|---|
| `center_error` | 정규화 횡오차 [-1,1], + = 우측 (없으면 NaN, `valid` 로 게이트) | control |
| `ema` | center_error EMA (`use_ema=True` 라 제어가 실제로 읽는 값) | control |
| `state` | `OK`/`LOW_CONF`/`OUTLIER`/`HOLD`/`LOST` | control (throttle) |
| `confidence` | **이산값**: 0.9(pair) / 0.5(coast) / 0.0(없음) | control (`conf_gate`) |
| `used_fallback` | coast(단일 차선) 사용 여부 | 기록 |
| **`n_corridors`** | **물리적으로 유효한 코리도어 수. `> 1` = 분기** | **(없음 — 판단 계층용)** |
| **`ego_rule`** | **무엇이 골랐나: `tracked`/`nearest`/`coast`/`branch_*`/`none`** | **(없음 — 판단 계층용)** |
| `heading` / `heading_label` | ego 중앙선 접선각(deg) | **아무도 안 씀** (C4/C5 만) |
| `curvature` / `has_curvature` | ego 중앙선 곡률 | **아무도 안 씀** |
| `left_conf` / `right_conf` | 0 또는 1 (binary) | **아무도 안 씀** |

제어(`control_core`)가 이 state 를 받아 조향·스로틀로 매핑한다 — 인지는 차량을 조작하지 않는다.

**`state` 는 이제 스로틀을 끊는다.** `LOST`/`HOLD` 는 물론이고 **`OUTLIER` 도** —
`throttle_outlier: 0.0`. OUTLIER 는 "지금 내가 주는 `ema` 를 믿지 마라" 는 뜻이고, 코리도어
전환 직후엔 부호까지 반대다. 145617 의 1.47초 역조향이 정확히 "인지는 알았는데 아무도 안
물었다" 였다.

**`n_corridors` / `ego_rule` 은 아직 아무것도 하지 않는다.** 분기가 실제로 보이는지, 그리고
무엇이 경로를 고르고 있는지를 **기록만** 한다 — 판단 계층은 측정 위에 설계한다. `ego_rule` 이
`nearest` 나 `branch_random` 이면 그것은 **결정이 아니라 자리표시자이고, 스스로 그렇게
말하는 것**이다. §8+++ 참조.

**7-label 분류(`W-L`, `YS-R`, ...)는 계약에 없다.** 디버그 오버레이 전용이고, 중앙선 도출에
쓰이지 않는다. 코리도어의 경로 정체성은 **색 조합**이 말한다 (흰-흰 = 본선, 노랑-노랑 =
노란 지름길; 흰-노랑은 페어가 될 수 없다 — §8+++).

## 6. 알려진 한계 — 0709 트랙 테스트 실측

`offline/0709 Track test(2025 Track)` 9세션 **3,004프레임** 로그 + 영상 분석 결과.
아래 4개는 **모두 front-view(§3)에서 비롯**한다 — 파라미터 튜닝으로 해결되지 않는다.

**실측 지표**

| 지표 | 값 |
|---|---|
| `valid` (ego 중앙선 확보) | 89.3% |
| ├ **pair** (좌우 쌍 성립, conf 0.9) | **34.9%** |
| └ **coast** (단일 차선 fallback, conf 0.5) | **54.4%** |
| 검출 실패 (conf 0) | 10.7% |
| `state` | OK 75.7% / **OUTLIER 13.6%** / LOST 9.2% / HOLD 1.5% |
| \|center_error\| 중앙값 | pair **0.140** vs coast **0.439** (coast의 38.7%가 >0.5) |

**(a) 라벨 turn 오분류 — 원근 수렴이 heading 부호를 지배**

`_build_instance`의 `turn`은 heading(상단 x − 하단 x) 부호로 정해지는데, front-view에선
**모든 차선이 소실점으로 수렴**한다. 좌측 차선은 위로 갈수록 우향(→ `R`), 우측 차선은
좌향(→ `L`)이라 **도로가 직선이든 곡선이든 좌측=R, 우측=L로 찍히는 구조적 편향**이 생긴다
(`heading_frac=0.06` → W=640에서 38px, 원근 수렴만으로 초과). 다만 turn 라벨은 **조향에
쓰이지 않는다**(§5) — 상위 미션 로직에만 영향.

**(b) 이종 페어링 — 중앙선이 엉뚱한 쌍에서 산출**

`lane_centers`의 페어 조건은 ①x_bottom 인접 ②side 상이 ③y겹침·최소간격뿐이다. **색도,
실선/점선도, 간격이 물리적으로 차선폭인지도 보지 않는다.** 그래서 내 차로 경계와 반대
차로 경계가 짝지어진다. front-view에선 간격이 y에 따라 변하므로 물리적 폭 게이트 자체가
불가능하다(`pair_gap_min=12.0`이 사실상 "0보다 크면 통과"인 이유).

**(c) coast 평행이동 — 상단으로 갈수록 발산**

`_shift`는 상수항만 옮긴다(= 모든 y에서 동일 픽셀 이동). 실제 차선폭은 하단에서 넓고
상단에서 좁은데 `width/2`(=160px)를 전 높이에 균일 적용하니 **상단 오차가 폭발**한다.
`_shift`는 원래 BEV(차선폭 상수)를 전제한 코드다. 파생 피해:
- coast의 \|center_error\|가 구조적으로 ≈0.5에 고정(위 표).
- `ego_center`의 near-lane 선택이 `x_bottom` 좌/우만 보므로 곡선에서 **좌↔우가 뒤집히며
  offset이 ±0.5로 반전** → 부호 반전 94회 중 **75회(80%)가 coast**,
  OUTLIER 410회 중 **358회(87%)가 coast**.
- `heading`도 이 오염된 중앙선 계수에서 뽑으므로 함께 오염.

**(d) center_error의 look-ahead가 매 프레임 달라짐**

offset을 평가하는 y는 pair면 두 차선의 y겹침 하단, coast면 검출 차선의 `ys.max()`다. 즉
**차선이 얼마나 길게 검출됐느냐에 따라 횡오차를 재는 지점이 앞뒤로 움직인다** — 제어
관점에선 look-ahead 거리가 프레임마다 바뀌는 것. 게다가 정규화 분모 `w/2`는 물리적 의미가
없다. 이 노이즈는 제어에서 증폭된다: `de=(e−prev_e)/dt`, dt≈0.067s(15fps)이므로 **미분항이
오차 점프를 15배 증폭**(e가 0.3 튀면 `kd·de`=0.45 = `steer_max`의 56%). 현재 `slew_rate=0.15`가
이를 겨우 막고 있다.

**(e) 그 외**

- **Tracker 출력 미사용**: `trk.update()`가 프레임간 EMA·차선폭 학습·L/R 지속성을 갖춘
  `trk.L`/`trk.R`을 계산하지만, `center_error`는 이를 쓰지 않고 매 프레임 `lanes`에서
  `ego_center()`를 새로 구한다. Tracker의 시간적 안정화가 출력에 반영되지 않는다.
- **분기(Y)**: MAX 병합이 분기를 분리 유지하고 방향 라벨로 구분하나, 어느 갈래로 갈지는
  상위 미션 로직의 몫.

## 7. 개선 로드맵 — 캘리브레이션 BEV 모듈

§6의 (a)(b)(c)(d)는 **캘리브레이션된 BEV 하나로 함께 풀린다**: 원근 수렴 소멸 → (a),
차선폭 픽셀 상수 → (c)의 `_shift`가 정당해지고 (b)의 물리적 폭 게이트가 가능,
고정 전방거리에서 cm 횡오차 → (d).

### 7.1 재사용 경계 (대회 트랙 이관 대비)

트랙마다 재작업하지 않으려면 **카메라 종속과 트랙 종속을 분리**해야 한다.

| 대상 | 종속 | 트랙 변경 시 | 카메라 재조정 시 |
|---|---|---|---|
| `K`, `D` (렌즈 왜곡) | 카메라 렌즈 | **재사용** | **재사용** |
| `H` (호모그래피), px/cm | 카메라 **장착 자세** | **재사용** | 재취득 (지면 사진 1장) |
| 차선폭·점선 규격·색 밴드 | **트랙** | 재측정 | 재사용 |

→ **BEV 모듈은 트랙을 모른다.** 호모그래피는 "픽셀 ↔ 지면 cm" 변환일 뿐이고 지면 평면
가정만 필요하다. 차선폭 등 트랙 정보는 그 위의 페어링·coast 로직이 쓴다.
설정도 이 경계로 쪼갠다 — **`camera.yaml`**(K·D·H·스케일·BEV 범위, 트랙 무관) +
**`track20XX.yaml`**(색 밴드·차선폭·검출 파라미터). 본대회에선 **track yaml만 새로 쓰고
camera 캘리브레이션은 그대로 재사용**한다.

### 7.2 모듈 구성

- **`dracer_core/calib.py`** (신규, 순수·ROS 무관): `CameraModel` — `undistort()` /
  `to_bev()` / `bev_coeffs_to_image()`(그리기용 역변환) / `px_to_cm()` / **`validate()`**.
  remap LUT는 캐시.
- **`offline/calibrate.py`** (신규 도구): 체커보드 N장 → `K`,`D`. 지면 체커보드 1장 → `H`·스케일.
  → `camera.yaml`. **사람이 점을 클릭하는 단계가 없어** 카메라를 만질 때마다 부담 없이 재실행.
- **`validate()` = 안전망**: 직선 구간 BEV에서 좌우 차선이 수직·평행하고 간격이 실측
  차선폭과 맞는지 자동 검사. 카메라 자세는 **미세 조정이 상시 일어나므로**, H가 낡으면 이
  검사가 먼저 실패해 *이상 조향의 원인이 캘리브레이션인지*를 즉시 가른다.
- **`perception_core` 통합은 `bev: CameraModel | None` 옵션**: `None`이면 현행 front-view
  경로 그대로, 넘기면 BEV 경로. warp 지점은 `LanePipeline.process`의 seam 주석 자리.
  **한 코드에서 A/B가 되고 롤백이 파라미터 한 줄.**

> **왜 체커보드인가** — 체커보드는 `K`·`D`(렌즈 고유, 장착 무관)를, 호모그래피는 장착 자세를
> 준다. **둘은 대체재가 아니다.** 왜곡을 걷어내지 않고 4점만 맞추면 왜곡이 호모그래피에
> 흡수되어 **화면 가장자리가 틀어지는데, 하필 coast가 그 가장자리에서 발생**한다. 또
> 수동 4점은 클릭 오차 ±2~3px가 원근으로 증폭되고(상단 1px ≈ 지면 수십 cm) 과결정 여유가
> 없다. 체커보드는 `cornerSubPix` 서브픽셀 + 코너 수십 개 최소제곱이다.

**방법**: 0709 녹화본(4패널)에서 원본 320×160을 복원해 **같은 프레임**에 두 파이프라인을

> **주의 — 녹화본 오염**: 패널1은 원본 위에 **ROI 사다리꼴 선(BGR 0,200,255 = 노란색)** 이

> 바꿨다(`recorder/raw/`, 무각인). 앞으로의 오프라인 재실행은 raw를 쓴다.

**신뢰성**: 복원 프레임의 front-view 재실행이 실차 CSV 실측과 거의 일치 →
복원이 원본을 충실히 재현하며, A/B가 유효하다.

| 지표 | front-view | BEV(근사) | 변화 |
|---|---|---|---|
| **OUTLIER** | 13.7% | **6.1%** | **−7.6%p (−55%)** |
| **\|Δce\| p95** (프레임간 점프) | 0.249 | **0.116** | **−53%** |

**해석**

- **검출률(valid·fail)은 거의 안 변한다** — 예상대로다. BEV는 검출을 늘리는 게 아니라
  **해석 정확도**를 고친다.
- **조향 부호가 뒤집히던 사례 확인**: `094912` f250(좌회전 곡선, 왼쪽 경계에 근접)에서
  front-view는 `ce=+0.44`(→ **좌조향**, 차선 쪽으로 파고듦), BEV는 `ce=−0.34`(→ **우조향**,
  이탈 회피). **front-view가 반대 방향 조향을 지시하고 있었다** — 095220 등의 코스 이탈이
  여기서 비롯됐을 가능성이 크다.
- **coast는 BEV로 해결되지 않는다** (54.4% → 51.3%). 곡선에서 반대편 차선이 **카메라 FOV 밖**
  으로 나가는 것은 좌표 변환의 문제가 아니다. 단 coast **오차**는 절반이 되어 훨씬 안전해진다.
  coast 빈도 자체를 줄이려면 **카메라 FOV·장착 각도**가 별도 논의 대상 → P4 이후로 미룸.

**★ 촬영 해상도를 320×160 → 640×480 으로 변경 (P2에서 발견, 계획 수정)**

당초 "주행과 동일한 파이프라인(320×160)으로 찍는다"였으나, **320×160에서는 지면에 눕힌
보드의 코너가 사실상 안 잡힌다.** 원근이 세로를 뭉개고 2:1 비등방 스케일이 한 번 더
뭉개서 칸이 7~11px까지 내려간다 — 합성 스윕에서 **24개 틸트/거리 조합 중 2개**만 검출됐다.

640×480으로 올리면 **near 18~22cm 구간에서 틸트와 무관하게 전부 검출**된다.
(near ≤15cm의 실패는 칸이 작아서가 아니라 **보드가 화면 밖으로 잘려서**다 — 칸은 오히려 48~70px.)

해상도 차이는 `CameraModel.rescale()`이 **정확히** 되돌린다(근사 아님). 두 해상도가 같은
센서 프레임의 순수 리스케일이므로 640×480 → 320×160은 네이티브 크기와 무관하게 정확히
(sx,sy)=(0.5, 1/3)이고, **K는 축별 선형 스케일 / D는 불변(정규화 좌표에 작용) / H는 `H @ S⁻¹`**.

### 7.6 P3 촬영 절차 · 입력 · 리스크

**필요 입력**: 체커보드 **한 칸 실측 크기(mm)** — 프린터 자동 축소를 감안해 **자로 잰 값**을
써야 한다(공칭 25mm를 그대로 넣으면 BEV의 cm 전체가 틀어진다). 인쇄물은 **평판에 부착**
(휘면 K·D가 통째로 오염되고 복구 불가). 트랙 **차선폭(cm)** — 페어링 게이트·coast·검증용.
점선 규격은 있으면 검증에 쓰되, 체커보드가 지면 자 역할을 하므로 필수는 아니다.

패턴: A4 / 6행 9열 / 25mm → **내부 코너 8×5** (짝수×홀수 → 회전 모호성 없음).
OpenCV 호출은 `(8, 5)`이지 6×9가 아니다.

**절차** (상세는 `capture_camera_calib.py` 도크스트링):
1. `vehicle_config.yaml`을 **백업**하고 `IMAGE_WIDTH/HEIGHT`를 640/480으로 → 카메라 재시작
2. **K·D**: 보드를 정면으로 들고 여러 각도·거리 20장. **화면 가장자리를 반드시 덮을 것**
   (왜곡이 가장 큰 곳이자 coast가 발생하는 곳). 웹(:5000)에서 **min gap** 확인
3. **H**: 보드를 노면에 평평히 눕히고 화면에 다 들어오게, 좌우 중앙에. **거리는 재지 않아도
   된다** — `ground_pose()`가 solvePnP로 산출한다(§7.7)
4. **해상도 원복** (안 하면 주행 인지가 640×480으로 돌아간다)
5. `offline/calibrate.py` → `camera.yaml`

**리스크 — 지면 평면 가정**: BEV는 노면이 평면임을 전제한다. 2025 트랙은 천/매트라
**주름**이 있고(예: `drive_20260709_095220` f150), 주름 위 차선은 평면을 벗어나 BEV 위치
오차가 된다. 본대회 노면이 매트라면 촬영 전 최대한 펴는 것 외에 알고리즘적 보정은 어렵다.

**`near_cm` 실측을 제거했다.** 보드가 지면 평면을 정의하므로 `ground_pose()`가 solvePnP로

선택**할 뿐이다. → **카메라 재조준 시 지면 사진 1장만 다시 찍으면 되고, 자를 댈 필요가 없다.**

**차선폭 35cm 확인**: 실측이 내부 32cm / 외부 38cm(테이프 폭 3cm)인데, 검출 다항식은
테이프의 **중심선**을 따라가므로(`sliding_window_lanes`가 창 내 픽셀 x를 평균) 코드가 쓰는
차선폭은 **중심선 간 35cm**다. BEV 시각화에서 35cm 가이드가 실제 차선과 일치해 교차검증됐다.

**★ coast 54%의 물리적 원인 규명 — 카메라 FOV**

> ⚠️ **이 절의 결론은 §7.9에서 반증됐다.** coast의 지배적 원인은 FOV가 아니라
> `sliding_window_lanes`의 **선행 miss 카운트 버그**였다. 그것을 고치자 같은 영상에서
> coast가 72% → 30%로 떨어졌다. 아래 FOV 수치 자체는 유효하나, "알고리즘으로는 못
> 푼다"는 단정은 **틀렸다.** 물리 한계를 선언하기 전에 알고리즘을 먼저 의심했어야 했다.

가시 지면이 **x=±22cm, y=21~55cm**뿐이다. 차선폭 35cm의 경계는 ±17.5cm이므로
**좌우 여유가 4.5cm에 불과하다.** 차량이 조금만 치우치거나 곡선에 들면 한쪽 차선이 즉시
FOV를 벗어난다 → §7.4에서 "BEV로도 coast가 안 줄어든다(54.4→51.3%)"고 한 것의 실체다.
전방도 **55cm**까지라 look-ahead가 짧다. 렌즈 화각을 넓히거나 카메라를 더 높이/뒤로 두면
개선되지만, **우선순위는 §7.9 이후로 낮아졌다.**

**BEV 격자**: 250×173px @ 5px/cm (x=±25cm, y=21~55cm). 차선폭 35cm = 175px(BEV 폭의 70%).

**① 프레임 드랍의 진범은 디버그 렌더링이었다.**

| 경로 | 비용 | 비고 |

**③ 해상도 불일치는 조용히 터지는 함정이었다.** `camera.yaml`(320×240)과 `vehicle_config`(320×160)이

**단, 0711 세션에서는 이게 발동하지 않았다** — 녹화된 raw가 320×240이라 보드는 이미 240으로 돌고

**④ 320×240으로 통일했다.** 근거는 4:3이 아니라 **비용이 0이기 때문**이다 — 검출 경로 비용이

동일하므로, 160은 **얻는 것 없이 세로 해상도만** 버리고 있었다(원거리 8cm가 10.4행 → 6.9행).
`match()` 덕분에 160으로 되돌리는 데 코드 변경이 필요 없다.

### 7.9 슬라이딩 윈도우 선행 miss 버그 (2026-07-11) — coast의 진짜 원인

D3-G에서 `perceive.launch`를 띄우자 마스크 패널에는 차선 두 줄이 **선명한데** 슬라이딩 패널은
`0 lane`, state는 계속 `LOST`였다. 윈도우 사각형이 **BEV 하단에만 3~4개** 그려지고 끝난 게 단서였다.

`sliding_window_lanes`는 창을 BEV **맨 아래 행부터** 위로 쌓는데, `miss`를 **첫 hit 이전부터**
세고 있었다. 하단이 비어 있으면 `sw_max_miss=3`에 걸려 **차선 픽셀에 도달하기도 전에 break** 된다.

```
win_h = 207//12 = 17
창 0: row 190..207  비어있음  miss=1
창 1: row 173..190  비어있음  miss=2
창 2: row 156..173  비어있음  miss=3 → break 💀
창 4: row 122..139  ← 차선은 여기 있었다. 도달 못 함.
```

**하단은 일상적으로 빈다.** 곡선이나 약간의 치우침만으로 차선이 카메라 좌우 FOV를 벗어나고,
BEV 바닥(26cm)은 애초에 이 렌즈가 차선을 볼 수 있는 높이가 아니다(양쪽 경계는 **~30cm부터**
보이고, 26cm에서의 가시 범위는 **-19.7~+2.0cm**로 심하게 비대칭이다). base peak가 "이 열에
픽셀이 있다"고 이미 증명했는데, 거기까지 **올라가는 과정**을 miss로 세어 스택을 죽이고 있었다.

→ **hit이 하나라도 있은 뒤의 공백만 miss로 센다.** `sw_min_hits`(2)와 `sw_min_span`(0.30·H)이
그대로 남아 있어 정지선 같은 가로 줄무늬는 여전히 기각된다(검증 완료).

0711 3세션 재측정 — **이번 작업 전체에서 가장 큰 이득이 여기서 나왔다:**

**교훈: 물리 한계를 선언하기 전에 알고리즘을 먼저 의심하라.** §6/§7.7은 coast 54~72%를 "카메라
FOV의 물리적 한계라 알고리즘으로는 못 푼다"고 단정했고, 그 결론이 **렌즈 교체를 별도 과제로

지배적 원인이라는 **인과 주장은 검증되지 않은 것이었다.**

그래서 값이 그 문턱 너머로 **지속 이동**하면 이후 모든 프레임이 같은 낡은 값과 비교돼 같은 이유로
거부된다 — **스스로를 영원히 재거부한다.**

그리고 이건 정당한 이벤트에서 터진다. 트래커가 보이는 경계를 좌↔우로 바꾸면 코리도어 중심이

## 8-. 기각된 접근 — 연속 confidence (2026-07-12)

**시도하지 마라. 측정했고, 틀렸다.**

### 동기 (여전히 타당하다)

`confidence` 는 `0.9`(쌍) / `0.5`(coast) / `0.0`(없음) 세 값뿐이다. `conf_gate 0.4` 와
`conf_low 0.25` 사이에서 이 값이 실제로 답할 수 있는 질문은 **"차선이 하나라도 있나"**
뿐이고, 그건 `valid` 플래그가 이미 답한다. **정보량이 0이다.**

그래서 파이프라인이 이미 계산해놓고 버리던 증거들을 모아 연속 품질 점수를 만들려 했다:
피팅 잔차(`rms`), 코리도어 세로 span, 페어 폭 오차, coast 지속 프레임(`pair_age`),
직전 EMA 와의 불일치(`agree`).

따라서 전환 직전 coast 값과 전환 직후 pair 값의 차이는 **그 coast 가 실제로 얼마나
틀렸는지에 대한 실측 오차**다. 0711 완주 3세션에서 이 전환이 20회 있었다.

| `pair_age` (작을수록 좋음) | **−0.242** | ✗ 반대 |
| `agree` (작을수록 좋음) | **+0.553** | ✓ 유일한 예측기 |
| 검출 차선 수 | −0.012 | · 신호 없음 |
| **합성 quality** | **+0.246** | ✗ **반대** |

오차 상위 4건의 실제 인자값:

```
   오차   span    rms   age   agree
 38.1cm   0.98   1.28     3   0.307
 29.7cm   0.94   1.16    19   0.058
 17.8cm   0.98   1.51     4   0.215
 16.9cm   0.98   0.98     0   0.005
```

### 왜 틀렸나 — 이게 배울 점이다

**38cm 틀린 coast 의 span 은 0.98 이고 피팅 잔차는 1.28cm 다. 기하학적으로 흠이 없다.**

당연하다. **틀린 coast 는 완벽하게 좋은 차선 피팅을 엉뚱한 쪽으로 민 것**이다. 기하는
티 하나 없고, 틀린 것은 **정체성**이다. 기하학적 청결도로 정체성 오류를 잡으려 한 것이
설계 오류였다. 두 질문은 다르다:

* "이 프레임의 기하가 깨끗한가" ← span·rms 가 답한다
* "이 프레임의 **답이 맞나**" ← 아무도 답하지 않는다

그리고 `pair_age` 는 **정확히 거꾸로**였다. 나쁜 coast 는 age 0~4, 즉 페어를 놓친 **직후**에
일어난다 — 전환 구간, 곡선, 정체성이 뒤집히는 바로 그 순간이다. 반대로 **긴 coast(age 31+)
는 오히려 정확하다** (완만한 구간이라 안정적). 실측: `pair_age` p50=31, p75=57, **max 112**.
**coast 는 이 트랙의 정상 모드다** (프레임의 50%; 렌즈 FOV 가 두 경계를 동시에 못 본다).
coast 를 벌주는 것은 오류가 아니라 **물리를 벌주는 것**이다.

### 그래서 결론

유일하게 예측하는 `agree`(시간적 불일치)는 **`_Stabilizer` 의 `outlier_jump` 가 이미
하고 있는 바로 그 검사**다. 그리고 `throttle_outlier`(2026-07-12)가 OUTLIER 에서 스로틀을
끊는다. **작동하는 유일한 신호는 이미 제자리에 있었다.** 합성 점수는 그 신호를 오도하는
네 개의 항으로 희석할 뿐이다.

게이트로 걸었다면: 심각한 오차(>15cm) 4건 중 **0건 포착**, 정상(<5cm) 12건 중 **4건 오탐**.
**완주하던 주행을 망가뜨렸을 것이다.**

### 다시 시도한다면

기하가 아니라 **정체성**을 겨냥해야 한다. 후보 신호:
* coast 가 미는 방향이 직전에 **확인된 페어**와 모순되는가 (adopt 가 이미 일부 처리)
* 트래커가 **사용하지 않는** 차선이 보이는가
* 합성된 반대편 경계 위치에 **실제로 마스크 픽셀이 없는가** ← 유령 경계의 직접 증거

특히 마지막이 유망하다. 145617 의 유령 R 은 `+43cm` 를 가리켰고 거기엔 차선이 **없었다**.
그건 BEV 마스크를 한 번 조회하면 알 수 있는 사실이다.

> **→ 그 가설이 맞았다. §8+ `coast_side` 참조.** 기하가 아니라 마스크에 물었더니,
> coast 좌/우 오류를 오탐 0 으로 잡아낸다.

**부수 소득**: 이 실험이 P2 의 음수 폭 버그를 노출시켰고(`ROLLBACK.md`), `pair_age`·`rms`
분포 실측치를 남겼다.

## 8--. 기각된 접근 — 쌍검출률을 소프트웨어로 올리기 (2026-07-12)

**쌍검출 상한(65/56/37%)은 진짜로 렌즈 FOV 다. 커밋 `2d329ee` 의 결론이 옳았다.**
찾아본 경로를 전부 적어둔다. 다시 파지 마라.

### 의심했던 것 ①: 베이스 히스토그램이 BEV 하단 절반만 본다

`hist = (mask[H//2:] > 0).sum(axis=0)` — 원거리에만 있는 차선은 피크가 안 잡혀 슬라이딩
윈도우가 시작조차 못 한다고 의심했다. **측정: 아무 효과 없다.**

| 베이스 히스토그램 | 쌍검출 | coast | valid | OUTLIER |
|---|---|---|---|---|
| 하단 50% (현재) | 62/54/34 | 50% | 100% | 1.1% |
| **전체 높이** | 61/53/33 | 50% | 100% | 1.1% |
| 하단 75% | 61/54/33 | 50% | 100% | 1.1% |
| 하단 50% + 전체 폴백 | 62/54/33 | 50% | 100% | 1.1% |

슬라이딩 윈도우는 피크 **열**에서 시작해 **맨 아래 행부터** 위로 쌓는다. 피크를 어디서
찾든 스택은 바닥에서 출발하므로, 히스토그램 창을 넓혀도 원거리 차선이 새로 살아나지
않는다. §7.x 의 "선행 miss 는 miss 가 아니다" 수정이 이미 그 등반을 허용하고 있다.

### 의심했던 것 ②: 슬라이딩 윈도우 게이트가 너무 빡빡하다

`sw_min_span`(13건) · `sw_min_hits`(2건) · `merge_dx` 병합(5건). **1,505프레임 중 20건.**
무시 가능하다.

### 의심했던 것 ③: 페어 게이트가 과잉 거부한다

**coast 프레임의 92~95% 는 애초에 차선이 하나만 검출된다** (2개 이상 검출: 5~8%).
그 5~8% 에서 두 차선의 실측 간격은:

```
p10 = -7cm (교차!)   p50 = 18~20cm   p90 = 25~46cm      (게이트 통과 범위 26~44cm)
```

**18~20cm 는 차선폭(35cm)이 아니다.** 같은 테이프가 쪼개졌거나 서로 교차하는 조각들이고,
페어 게이트는 **옳게 거부하고 있다.** (`pair_width_tol` 을 넓히면 이것들을 코리도어로
받아들이게 된다 — 그건 개선이 아니라 회귀다.)

### 의심했던 것 ④: 반대편에 픽셀이 있는데 못 잡는다

커밋 `2d329ee` 의 "단일차선 프레임에서 반대편 마스크가 비어 있는 비율 96/96/75%" 에서
남는 25% 를 쫓았다. 실제로 반대편에 픽셀은 **거의 항상 있다(99%)**. 그러나:

```
frame 69 (145617):  반대편 픽셀 2,929개 —  상단 2,049 / 하단 880
                    하단 880 개가 160 열에 흩어져 열당 5.5 개
                    sw_peak_min = 10 -> 피크 미달
                    (그리고 White 마스크는 아예 비어 있다)
```

**응집된 테이프가 아니라 원거리의 흩어진 픽셀**이다. 차선이 아니다.

### 결론 (⚠ 부분 정정됨 — §8++ 참조)

**2차선 프레임에 대해서는 맞다.** coast 는 소프트웨어 결함이 아니라 **이 렌즈가 두 경계를
동시에 보지 못한다**는 사실이다 (근거리 26cm 에서 횡방향 시야 -19.7~+2.0cm; 양쪽 경계는
~30cm 부터 보인다). **렌즈·장착 과제**로 남는다.

**그러나 3차선 이상 프레임(6%)에 대해서는 틀렸다.** 위 ③ 에서 "인접 쌍의 간격이 18~20cm 라
같은 테이프가 쪼개진 것" 이라고 결론냈는데, 그게 아니었다 — **로터리의 노란 선이 흰색 차선
경계 5~10cm 옆을 지나고**, 흰색/노란색은 슬라이딩 윈도우가 따로 돌려 병합되지 않는다.
그래서 `[흰, 노란, 흰]` 으로 잡히고, **인접 쌍만 짝짓는 페어링이 진짜 코리도어 `(흰, 흰)`
= 35cm 를 후보에 올리지도 못했다.** §8++ 에서 고쳤다 — 쌍검출 +3%p, OUTLIER 절반.

## 8---. 기각된 접근 — 핫패스 "명백한" 최적화 (2026-07-12)

**둘 다 측정했고, 둘 다 느려졌다.** 그리고 애초에 최적화할 필요가 없다.

### 스테이지별 실측 (145617, 463프레임, macOS)

```
    색 검출 (cvtColor + 마스크, src_rows 밴드)    72 µs
    BEV 워프 (remap x2)                          42 µs
    morph_gate                                   25 µs
    sliding_window (Python 루프)                431 µs   <- 파이프라인의 60%
    ------------------------------------------------
    process() 전체                             ~720 µs   (예산 33ms @30Hz)
```

### 시도 ①: `cv2.inRange` 로 numpy 불리언 체인 대체 → **19% 느려짐**

| | |
|---|---|
| 구: `split` → 비교 4회 → `&` 3회 → `astype` → `*255` | **53.8 µs** |
| 신: `cv2.inRange` x2 | 64.3 µs |

"C 루프 한 번이 numpy 임시배열 예닐곱 개보다 빠르다" 는 직관이 틀렸다. 구코드는
`cv2.split` 을 **한 번** 하고 H/S/V 평면을 **두 색이 공유**한다. `inRange` 는 3채널 전체를
**두 번** 훑는다. 밴드가 작아서(148x320) numpy 오버헤드보다 중복 순회가 더 비싸다.

### 시도 ②: 마스크 2장을 2채널로 쌓아 `remap` 1회 → **15% 느려짐**

| | |
|---|---|
| 구: `to_bev(mw)` + `to_bev(my)` | **41.9 µs** |
| 신: `to_bev(merge(mw,my))` + `split` | 48.4 µs |

BEV 가 232x207 로 작아서 remap 자체가 싸다. 아낀 두 번째 remap 보다 `merge` + `split` 이
더 비싸다. (LUT 조회를 두 번 하는 게 손해라는 직관 — 캐시에 다 들어간다.)

### 결론

**성능 문제가 없다.** 0.72 ms / 33 ms 예산이고, 보드는 이미 30.0Hz 를 **드랍 0** 으로
낸다(1,502 프레임, 최대 갭 38ms). 유일하게 무거운 곳은 sliding_window 의 Python 루프
(431µs, 60%)인데, 그건 numpy 호출 오버헤드(창 72개 x 호출 6회)이지 데이터량이 아니다 —
고치려면 창 등반 전체를 벡터화해야 하고, 지금 아무도 그걸 필요로 하지 않는다.

**측정 없는 최적화는 회귀다.** 렌더링을 핫패스에서 뺀 것(§7.8)이 진짜 최적화였고, 그건
4배짜리였다. 이 둘은 -19% 와 -15% 였다.

## 8++. 채택된 접근 — 모든 쌍 페어링: 분기가 보이지 않으면 선택할 수 없다 (2026-07-12)

### 분기점에서 다중 코리도어가 나오는가? — **나오지 않았다. 코드가 버리고 있었다.**

7,785 프레임(대시캠 8클립 + 성공주행 3세션)을 측정한 결과:

```
검출 차선 수:  1개=42%   2개=52%   3개=5%   4개=1%
유효 코리도어: 0개=47%   1개=53%   2개 이상 = 0 프레임   ★
```

**차선은 3~4개가 보이는데(6%) 코리도어는 단 한 번도 2개가 안 나온다.** 로터리 분기 판단의
전제가 무너진다 — 보이지 않는 갈림길은 선택할 수 없다.

### 원인: 인접 쌍만 짝짓기 + W/Y 를 별도 슬라이딩 윈도우로 검출

`lane_centers` 는 x 정렬 후 `zip(ls, ls[1:])` — **인접 쌍만** `_pair_gate` 에 넘겼다.

그런데 로터리에서는 **노란 선이 흰색 차선 경계 5~10cm 옆**을 지난다. 흰색과 노란색은
슬라이딩 윈도우가 **따로** 돌고 `_same_lane` 병합도 **색 안에서만** 일어난다 (그리고 그게
맞다 — 실측 결과 그 근접 교차색 쌍의 **88% 는 진짜로 다른 테이프**다).

그래서 프레임이 `[흰, 노란, 흰]` 으로 잡히고:

```
    (흰, 노란) =  5cm  -> 거부
    (노란, 흰) = 30cm  -> 거부
    (흰,   흰) = 35cm  -> 후보에 오르지도 못한다   ★ 진짜 코리도어
```

실측 (차선 3개 이상인 323 프레임):

| | 코리도어 2개 이상 |
|---|---|
| 인접 쌍 (구) | **0 프레임** |
| 모든 쌍 (신) | **158 프레임** |

### 수정: `itertools.combinations(ls, 2)`

후보 집합만 넓힌다. `_pair_gate`(겹침 ≥30%, 간격 ≥8cm, **폭 35±25%**)는 그대로다 —
**수용 기준이 아니라 후보 범위를 고친 것이다.** 인스턴스가 최대 6개(색당 3개)라 게이트
호출은 최대 15회, 비용 변화 0 (0.72 ms/frame 그대로).

### 결과

| | 쌍검출 | coast | OUTLIER | 부호반전 | 진동 | **코리도어 2+** |
|---|---|---|---|---|---|---|
| **성공주행 3세션** 구 | 50% | 50% | 1.1% | 3 | 0 | **0.0%** |
| 신 | **53%** | 47% | **0.6%** | 3 | 0 | **4.0%** |
| **대시캠 8클립** 구 | 51% | 49% | 0.7% | 9 | 0 | **0.0%** |
| 신 | **54%** | 46% | **0.5%** | **7** | 0 | **2.8%** |

`valid 100% / LOST 0%` 유지. 145515·145537 은 **OUTLIER 가 0% 가 됐다.**

### ★ 그리고 분기에서 무엇이 선택하고 있었나

```
분기 프레임 (코리도어 2개 이상): 218 (3.0%)   최대 4개
분기 구간: 28회, p50 = 6프레임, max = 20 (0.7초)

분기에서 ego 를 고른 규칙:
    tracked:  213 (98%)   <- 이전 프레임의 코리도어를 그냥 유지한다
    nearest:    5 ( 2%)
```

**분기에서 시스템은 선택하지 않는다. 이어갈 뿐이다.** 트래커의 시간적 정체성이 98% 관통한다.
차선 유지로는 합리적인 기본값이지만, **차가 노란 지름길을 절대 못 탄다**는 뜻이다.

→ **판단 계층의 역할이 정확히 이것이다: 분기에서 `tracked` 규칙을 오버라이드하는 것.**
   그리고 **0.7초 안에** 해야 한다.

`LaneState.n_corridors` / `LaneState.ego_rule` 로 발행·기록한다. **아직 아무것도 하지
않는다** — 판단 계층은 측정 위에 설계한다. `ego_rule: nearest` 는 결정이 아니라
**자리표시자이고, 스스로 그렇게 말한다.**

## 8+++. 채택된 접근 — 색은 경로다, 평행도는 "내가 그 안에 있는가"다 (2026-07-12)

§8++ 로 분기가 보이게 됐다. 그런데 **보이는 것이 전부 진짜는 아니었다.**

### 만들어지던 코리도어의 색 조합

| | 전체 | ego 로 사용 | **분기 프레임에서** |
|---|---|---|---|
| 흰-흰 | 72.8% | 78.5% | 30.9% |
| 노랑-노랑 | 21.0% | 20.5% | 26.4% |
| **흰-노랑** | 6.2% | 0.9% | **42.7%** ★ |

**색 조합이 곧 경로 정체성이다.** 본선은 흰-흰, 노란 지름길은 노랑-노랑. **흰-노랑은
물리적으로 한 차선의 두 경계일 수 없다** — 두 경로의 경계가 교차점에서 우연히 35cm
떨어진 것이다. `pair_width_tol` 로는 못 잡는다: 둘 다 진짜로 35cm 다.

그리고 평상시엔 무해하지만(ego 의 0.9%), **분기에서는 코리도어의 43%** 다. 판단 계층이
동작해야 하는 바로 그곳이다.

→ **`pair_same_color: true`.** 흰-노랑 ego 35프레임 → **0**.

### ⚠ 그리고 평행도로 게이트를 걸었다가, 갈림길을 통째로 지울 뻔했다

`_pair_gate` 는 간격의 **평균**만 봤다. 벌어지는 두 선도 평균이 35cm 면 통과한다. 그래서
평행 게이트(`gaps.max() - gaps.min() <= 8cm`)를 넣었다. 안정성은 좋아졌다(반전 8→7).

**그런데 노란 갈림길이 사라졌다.** 노란 분기 프레임(노란 차선 3개 이상, 40프레임)에서:

| 평행 게이트 | 코리도어 2개 이상 |
|---|---|
| 없음 | **52%** |
| 15cm | 2% |
| **12cm 이하** | **0%** |

파고들어 보니:

```
노란차선 | 코리도어A 평행/폭  | 코리도어B 평행/폭  | 경계 공유?
      3 |   1.1cm    34cm  |  32.7cm    28cm  | 예
      3 |   4.2cm    33cm  |  20.5cm    27cm  | 예
      3 |   5.1cm    34cm  |  16.7cm    29cm  | 예
      ...
★ 두 코리도어가 경계를 공유: 21/21   가짜 교차쌍: 0/21
```

**21/21 전부 `[A,B]` + `[B,C]` — 경계를 공유하는 진짜 갈림길이다.** 그리고 **두 번째
코리도어가 벌어지는 게 당연하다: 그게 갈림길이니까.** 갈라져 나가는 경로는 우리 시야에서
벌어져 보인다.

```
    코리도어 A   평행 3.2cm,  폭 34cm   <- 내가 지금 달리는 차선
    코리도어 B   벌어짐 32.7cm, 폭 28cm  <- 아직 안 들어간 다른 경로
```

**평행도는 "이것이 코리도어인가" 가 아니라 "내가 이 안에 있는가" 다.**

→ `_pair_gate` 는 `spread` 를 **보고만 한다**. `lane_centers` 는 전부 남긴다
  (`n_corridors` 가 갈림길을 센다). `ego_center` 만 벌어지는 것을 "지금 달리는 차선" 으로
  치지 않는다. 게이트로 썼으면 판단 계층이 판단할 것이 없어졌을 것이다.

### 결과 (대시캠 8클립 + 성공주행 3세션, 7,172프레임)

| | 쌍검출 | OUTLIER | 부호반전 | 진동 | 분기 2+ | ego 색 |
|---|---|---|---|---|---|---|
| 구 (색·평행 무시) | 54% | 0.5% | 10 | 0 | 3.0% | WW=3021 YY=790 **WY=35** |
| 신 | 53% | **0.3%** | **7** | 0 | 1.5% | WW=3035 YY=793 **WY=0** |

분기 3.0% → 1.5% 는 **가짜 흰-노랑 코리도어가 빠진 것**이다. 노란 갈림길은 **52% 그대로**.
0711 재생: 145515·145537 **OUTLIER 0%**, valid 100%, LOST 0%.

### `branch_policy` — 판단 계층의 자리, 그리고 그것이 자리표시자라는 고백

```
policy      부호반전  진동  OUTLIER   분기에서 선택
 keep (기본)     7     0    0.3%    tracked=105 nearest=5
 nearest         7     0    0.3%    tracked=91  branch_nearest=17
 random          8     0    0.5%    tracked=90  branch_random=17
```

**반드시 래치한다.** 프레임마다 새로 고르면 차선폭만큼 진동한다 — 가설이 아니라 측정이다
(프레임 단위 `coast_side` flip 이 진동 0 → 3 을 만들었다, §8+). 분기 **진입 시 한 번**
고르고 `Tracker.adopt` 로 밀어넣어 정체성이 유지되게 한다. **랜덤도 진동하지 않는다(0).**
반전 7→8 은 **진짜 경로 변경 17회**이지 떨림이 아니다.

기본값 `keep` 은 **현재 동작 그대로**다 (트래커가 이어간다 = 차는 노란 지름길을 절대 못
탄다). `random` / `nearest` 는 실험용이고, `LaneState.ego_rule` 이 `branch_random` 이라고
**스스로 말한다.** 결정이 아니라 자리표시자다.

## 8+. 채택된 접근 — `coast_side`: coast 의 좌/우를 마스크로 반증한다 (2026-07-12)

§8- 이 가리킨 가설의 검증과 채택. **기하가 아니라 마스크에 물었다.**

### 원리

coast 는 경계 **하나**를 보고 "코리도어는 이것의 왼쪽/오른쪽에 있다" 고 **주장**한다.
그 주장은 트래커의 정체성이고, 틀리면 코리도어 중심이 **차선폭만큼** 어긋난다 — 완벽한
차선 피팅, span 0.98, 잔차 1.3cm 를 달고서. 기하로는 절대 못 잡는다 (§8-).

그러니 기하가 못 묻는 것을 묻는다: **우리가 주장하는 자리에 유령 경계를 놓고, 거기를 본다.**
합성한 반대편 경계 아래에 마스크가 없는데 **거울상 가정 쪽에는 있다면**, 우리가 방향을
잘못 골랐다.

### 검증 (0711 완주 3세션, coast → pair 전환 20건)

전환 직후의 pair 는 두 경계를 실제로 보고 만든 값이니 믿을 수 있다 → 직전 coast 값과의
차이가 **그 coast 의 실측 오차**다.

| | 거울상 근거 > 0 |
|---|---|
| 오차 < 15cm 인 coast **16건** | **0건** ← 오탐 0 |
| 오차 > 15cm 인 coast 4건 | **2건** (18cm, 29cm) |

`corr(거울상 근거, 오차) = +0.486`. §8- 의 어떤 기하 항보다 강하다.

### 세 개의 가드 — 하나라도 빠지면 위험하다

1. **`coast_flip_support ≥ 0.15`** — 거울상에 실제 근거가 있어야 한다.
   (전체 coast 프레임에서 거울상 근거 평균 **0.005** — 이 신호는 원래 희귀하다.)
2. **`coast_flip_empty ≤ 0.05`** — 우리 유령은 근거가 없어야 한다. 없으면 `s_alt > s_now`
   가 0.35 대 0.30 같은 동전던지기에서 뒤집는다. **오탐 flip 은 나쁜 프레임이 아니라
   나쁜 트래커다** (`reseat_coast` 가 영구화한다).
3. **축 가드 — flip 은 코리도어를 차량 축 쪽으로 당겨야 한다.** ★ **이게 로터리를 지킨다.**
   마스크는 "저기 차선이 있다" 는 말할 수 있어도 **"저게 내 차선이다"** 는 못 한다.
   2차선에서는 같은 말이지만 4차선 로터리에서는 아니다: 1차선에서 coast 중일 때 내 쪽
   경계는 FOV 밖(`s_now=0`)이고 2차선의 바깥 경계는 잘 보일 수 있다 → **옆 차선으로
   flip**. 미션의 핵심 구간에서 원치 않는 차선 변경이다.
   그런데 **차는 그 코리도어 안에 있다.** flip 이 중심을 축에서 **멀어지게** 하면 그건
   옆 차선이다. 뺄셈 한 번이면 구별된다.

   실측: 축 가드가 해로운 flip 을 정확히 걸러냈다. 없을 때 문턱 0.15 에서 상황이 **악화**
   (오차 8.5→9.1cm), 있을 때 **개선** (8.5→7.6cm). 안정 구간도 `[0.25]` 한 점에서
   `[0.10, 0.20]` 세 점으로 넓어졌다 (= 과적합 위험 감소).

4. **`reseat_coast` — 교정은 트래커를 고쳐야 한다.** 출력만 뒤집으면 다음 프레임에
   (여전히 틀린) 정체성이 또 잘못 coast 하고 마스크가 또 뒤집는다 → **차선폭만큼 매
   프레임 진동**. 실측으로 확인했다: 출력만 뒤집었을 때 진동 0 → **3**. `adopt` 과 같은
   교훈이다.

### 결과 (0711 3세션 1,505프레임)

| | OFF | ON |
|---|---|---|
| coast 실측 오차 mean | 8.5cm | **7.6cm** |
| p90 | 19.3cm | **15.4cm** |
| \> 15cm 오차 | 4건 | **3건** |
| OUTLIER | 1.1% | 1.1% |
| 진동 | 0 | **0** |
| valid | 100% | 100% |
| **발동 횟수** | — | **1,505 프레임 중 1회** |
| 비용 | 0.64 ms/frame | 0.70 ms/frame |

**아주 드물게 발동하고, 발동할 때만 고친다.** 회귀 없음.

### 한계 (정직하게)

* **좌/우 검사이지 일반 coast 검증기가 아니다.** 15cm 오차 1건은 유령에 근거가
  충분했는데도 틀렸다 — **폭** 오류이지 **방향** 오류가 아니다. 다른 버그다.
* 남은 36cm 오차 1건은 양쪽 다 근거가 없어 **정보가 없다**. 못 잡는다.
* n=2 양성 사례. 축 가드가 문턱 민감도를 크게 줄였지만, **작은 표본이라는 사실은 변하지
  않는다.** `coast_flip_support: 0.0` 으로 끌 수 있다 (`ROLLBACK.md`).
